Test Environment



Ideally a penetration test should be run against a Test Environment
While testing against the Test Environment is ideal, most of the penetration tests occur against Production Environments.

What is a Test Environment
Some organizations have a:
• Test Environment in which changes and business functionality are evaluated before being applied in production.
• Development Environment, in which new code is created before being moved into the Test Environment

Why run a penetration test against a Test Environment
Ideally a penetration test should be conducted against a Test Environment. After all, it was designed for testing and any system or service crash in a Test Environment should not impact production processing

Why in the Real world run a penetration test against a Test Environment is not often possible
• Most organizations do not have a Test Environment
• Even for the organizations that have a Test Environment:
◇ could be not accessible
◇ could have major or subtle differences with the Production Environment that might invalidate our findings

What if the organization not have a Test Environment
If we have to testing against a production environment, first we should test the exploit inside a VM.
The VM have to run the same version of software as your target. We need to monitor the attack carefully with a tool like Wireshark and explore the effects of your attack on the VM afterward.
If we are running the attack against the client’s Test environment, then wireshark may not be necessary.
