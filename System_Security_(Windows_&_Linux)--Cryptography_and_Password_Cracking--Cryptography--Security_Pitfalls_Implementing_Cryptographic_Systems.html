Security Pitfalls Implementing Cryptographic Systems



Attacks against Implementations


Most of the times, an attacker will not directly attack the cryptographic algorithms. They instead attack their implementation.
How implement a cryptographic algorithm correctly:
• Not destroying plaintexts after use
• Not dealing with decrypted data( if we do, we have to do it carefully)
example: A system using temporary files to avoid data loss, might leave plaintext or decrypted data or both in the temporary file
• System using more than 1 key, should take care of all keys equally, because a single key leak renders the complete system useless.
• Allowing recovery of old keys can also act as a weak point
• ...

Attacks against Password


Many systems break because they rely on user-generated passwords that usually are not very strong.
Dictionary attacks work very well when the dictionary is targeted to the environment, country, age and language of the target

Attacks against Trust Models


Usually systems use Trust modules(like seen for the Public Key Infrastructure) to verificate that the system they are communicate with is trustable.
Attackers can exploit trust that the target give to other systems.
examples:
e-mail program: unless the public keys are certified by a trusted source (and unless that certification can be verified), the system is vulnerable
also if it use crypto algorithms to encrypt messages
website: sometime trust data for a website come from a database, what if we gain access to the database?

Attacks on the Users


Users are the weakest link of the chain.
With Social Engineering in combination with phishing websites or installed malware; cryptography or password strength are useless.
