3. Containment


Goal of Conatinment
The goal of containment is to keep the problem from getting worse and stop the bleeding.

Make first a copy of the unaltered system
As soon as the handler starts to recover the system, there is a point in which the evidence starts to become contaminated. If at all possible, the tile system image creation should come before this point so there is a copy of the unaltered system.

Containment includes three sub-phases
• Short-Term Containment → just to stop the damage
• Evidence Collection
• Long-Term Containment → make sure the bad guy is denied access


Survey the Situation
Deploy a small on-time team(usually the Identification Team) to survey the situation
You can also use as reference survey forms at: https://www.sans.org/information-security-policy/?category=incident-handling

Characterize the Incident
The FIRST organization distributes an incident Case Classification document(https://www.first.org/resources/guides/csirt_case_classification.html) that recommends characterizing an incident based on three areas:
• Incident Category:
◇ Denial of Service, Compromised Information, Compromised Asset, Internal Hacking, External Hacking, Malware, Policy Violations
• Criticality of impacted systems and data (affects response time)
• Sensitivity (affects who should be notified)

CyberCPR Tool
Hardened web application app (locally or with a cloud provider) that tracks incidents, systems, and evidence.
It supports the incident handler by providing a central repository for evidence found and protects legal admissibility by encrypting and hashing (MDS and SHA256) all data uploaded. This provides a secure alternative to emailing evidence and updates around the network.
◇ Enforces need-to-know on incidents
◇ All files are hashed and encrypted upon upload
◇ Tracks user tasks and activity
◇ Tracks attacker campaigns
◇ Automates key analysis
◇ Secure real-time OOB chat
