3. Project Scope


Project Scope: which are the concerns of the target organization?


Usually the Request for Proposal(RFP) issued by a client company provides limited details about the penetration test, because of that define clearly the Project Scope is very inportant.

• Project Scope and Rules of Engagement, which come first?
There is not a written rule, whatever target personnel are more comfortable with should come first
Setting first the Project Scope can helps to define Rules of Engagement.
Other times, the Rules of Engagement is already established by the client.
These two phases may be iterative

Scope for the test


• What systems to be tested?
◇ Domain names
▪ Individual Web Applications
◇ Network address ranges
▪ Individual hosts
• What systems should be avoided?
◇ particularly sensitive elements of the organization
• If others systems are discovered within the network address ranges or domain names that weren't considered in the original formulation of the scope, check with the target organization if they have to be tested
• Third Parties.
◇ Get written permission to test the equipment of any third parties
▪ Internet Service Providers(ISP): routers, switches, mail servers, and DNS servers
▪ Web Hosting companies: a single web server may house the web presence for a dozen or more customers. Attention!
◇ The target organization is responsible for getting the written permission

Biggest Security Concerns


Ask to the target organization about their biggest security concerns. What worries them? It is better to ask than guess!
List(not comprehensive) of Security Concerns and the organizations more impacted about them:
◇ Disclosure of sensitive information
▪ Financial services institutions
▪ Healthcare companies
▪ Educational organizations
▪ ...
◇ Interruption of production processing
▪ Manufacturers
▪ Financial services institutions
▪ Law firms
◇ Embarrassment due to defacement of website
▪ Advertisers

▪ ..
◇ Compromising of a machine to use as a jump-off point for deeper penetration(pivoting)
▪ Every type of organization can be significantly damaged by this issue

Avoid Scope creep


• Discuss threats, risks, and already-known vulnerabilities
◇ This is a kind of brainstorming session
◇ Discuss how to best test these areas of concern
• Focus on the area of concern
◇ Avoid a Scope creep
▪ With “Scope creep” we meaning the continuous and uncontrolled growth in a project scope. This can happen when the scope of a project is not properly defined.
If a “Scope creep” occur the entire penetration test can begin to be both costly of time and and money
◇ If focus is difficult because there are many area of concern, suggest to include in the test the low-hanging fruit to start

Denial-of-Service allowed or not


• Denial-of-Service attack (attacks explicitly designed for denial of service):
◇ Check only the version number of the target service and check in a list if could be vulnerable to Denial of Service
◇ Launch the Denial of Service attack against the the target running service. If the service died is vulnerable to Denial of Service BUT this potentially devastating impact on a production environment! Anyway if a service can be crashed by a remote attacker, it may be better to learn about it during controlled circumstances of a penetration test rather than waiting for an hacker to hit it.
• Others dangerous exploits:
◇ Some exploits could potentially crash a target service by trying to spawn a shell for the attacker by doing a buffer overflow exploit on the target.
Because such techniques manipulate memory they could render a service inaccessible
◇ There is always the possibility during a penetration test to crash a service/system:
◇ Even with no dangerous checks/attacks:
▪ the tester's tools will generate some strangely formatted packets that could cause a particularly feeble system to crash.
▪ Moreover the additional traffic load could push a highly loaded network's performance over its limit
Because of that there is always a possibility of a crash in a target system. What we have to do is minimize that chance and monitoring services to make sure that they continue to run while a test is underway.

Duration of a Penetration Test


Penetration tests typically last 1 to 3 weeks, followed by reporting interval that could last 1 week or more
• Some penetration test are longer; some are shorter
• To be efficient is best to have a person conducting manual tests in parallel to automated tools running
